# JUCE Programming Expert

This skill provides comprehensive guidance for developing with the JUCE framework, a cross-platform C++ application framework for creating desktop and mobile audio applications, including audio plugins and GUI apps.

## Core Principles

### 1. **JUCE Architecture Understanding**
- JUCE uses a modular architecture with 23+ modules (juce_core, juce_audio_processors, juce_gui_basics, etc.)
- Each module is a self-contained unit with specific functionality
- Modules must be linked with `PRIVATE` visibility to avoid ODR violations
- The framework supports VST3, AU, AUv3, AAX, LV2 plugin formats and standalone apps

### 2. **Project Setup Approaches**
JUCE projects can be managed two ways:
- **CMake** (recommended): Modern, flexible, better for CI/CD
- **Projucer**: JUCE's GUI-based project generator (supports Xcode, Visual Studio, etc.)

For CMake projects, always use:
- CMake 3.22 or higher
- C++17 standard
- `juce_add_gui_app()`, `juce_add_console_app()`, or `juce_add_plugin()` functions

## Common Development Tasks

### Creating a New JUCE Plugin

When helping users create plugins, follow this structure:

```cmake
# Minimal plugin CMakeLists.txt template
cmake_minimum_required(VERSION 3.22)
project(MyPlugin VERSION 0.1.0)

add_subdirectory(JUCE)  # or use find_package(JUCE CONFIG REQUIRED)

juce_add_plugin(MyPlugin
    PRODUCT_NAME "My Plugin"
    COMPANY_NAME "Your Company"
    PLUGIN_MANUFACTURER_CODE Manu
    PLUGIN_CODE Mplu
    FORMATS VST3 AU Standalone
    IS_SYNTH FALSE
    NEEDS_MIDI_INPUT TRUE
)

target_sources(MyPlugin PRIVATE
    PluginProcessor.cpp
    PluginEditor.cpp
)

target_compile_definitions(MyPlugin PUBLIC
    JUCE_WEB_BROWSER=0
    JUCE_USE_CURL=0
    JUCE_VST3_CAN_REPLACE_VST2=0
)

target_link_libraries(MyPlugin PRIVATE
    juce::juce_audio_utils
)
```

### Audio Processing Best Practices

1. **processBlock() is real-time safe**:
   - No memory allocation
   - No file I/O
   - No locks (if possible)
   - No std::cout or logging

2. **Use JUCE's DSP module** for common operations:
   - `juce::dsp::ProcessorChain` for effect chains
   - `juce::dsp::StateVariableFilter` for filtering
   - `juce::dsp::Gain` for gain staging
   - `juce::dsp::Reverb` for reverb effects

3. **Parameter handling**:
   - Use `AudioProcessorValueTreeState` for parameter management
   - Attach parameters in constructor
   - Use atomic values or message threads for thread-safe communication

### GUI Development with JUCE

1. **Component hierarchy**:
   - All UI elements inherit from `juce::Component`
   - Use `addAndMakeVisible()` to add child components
   - Override `paint()` for custom drawing
   - Override `resized()` for layout management

2. **Modern layouts**:
   - Use `juce::Grid` for grid-based layouts
   - Use `juce::FlexBox` for flexible box layouts
   - Use `Rectangle::reduced()`, `removeFromTop()`, etc. for manual layouts

3. **Look and Feel**:
   - Override `juce::LookAndFeel` methods for custom styling
   - Set with `component.setLookAndFeel(&customLookAndFeel)`
   - Remember to manage LookAndFeel lifetime carefully

### Module Selection Guide

Refer to **modules-reference.md** for detailed module descriptions and when to use each module.

Common module combinations:
- **Basic audio plugin**: juce_audio_processors, juce_audio_utils
- **GUI application**: juce_gui_basics, juce_gui_extra, juce_graphics
- **DSP processing**: juce_dsp, juce_audio_basics

## CMake API Quick Reference

### Essential Functions

1. **juce_add_plugin()**: Create an audio plugin target
2. **juce_add_gui_app()**: Create a GUI application
3. **juce_add_console_app()**: Create a console application
4. **juce_add_binary_data()**: Embed resources (images, audio files)
5. **target_link_libraries()**: Link JUCE modules (always use PRIVATE)

### Important Options

For detailed CMake options, see **cmake-options.md**.

Key options to remember:
- `FORMATS`: Plugin formats to build (VST3, AU, AAX, etc.)
- `COPY_PLUGIN_AFTER_BUILD`: Auto-install plugins after build
- `BUNDLE_ID`: Unique identifier (com.company.product)
- `PRODUCT_NAME`: Display name
- `PLUGIN_MANUFACTURER_CODE`: 4-char company ID
- `PLUGIN_CODE`: 4-char unique plugin ID

## Platform-Specific Considerations

### macOS/iOS
- **Code signing**: Required for plugins and iOS apps
- **Info.plist**: Auto-generated by JUCE CMake
- **Permissions**: Set via CMake options (MICROPHONE_PERMISSION_TEXT, etc.)
- **Universal binaries**: Use `-DCMAKE_OSX_ARCHITECTURES=arm64;x86_64`

### Windows
- **Visual Studio**: 2019 or later required
- **Windows 10+**: Minimum deployment target
- **WebView2**: Enable with `NEEDS_WEBVIEW2=TRUE` if using web browser

### Linux
- **Dependencies**: See `/docs/Linux Dependencies.md` in JUCE repo
- **Package managers**: Use apt, yum, etc. to install dependencies
- **Webkit**: Enable with `NEEDS_WEB_BROWSER=TRUE`

## Debugging and Testing

### Common Issues and Solutions

1. **Linking errors with modules**:
   - Always use `PRIVATE` visibility when linking JUCE modules
   - Ensure module is added before targets that use it

2. **Plugin not loading in DAW**:
   - Check plugin codes are unique (4 characters)
   - Verify format is enabled (VST3, AU, etc.)
   - Check code signing on macOS

3. **Build errors**:
   - Verify CMake version >= 3.22
   - Check C++17 is enabled
   - Ensure all dependencies are installed

### Testing Strategies

- Use JUCE's `UnitTest` class for unit testing
- Test plugins in multiple DAWs
- Use Projucer's AudioPluginHost for quick testing
- Enable JUCE's `JUCE_DEBUG` mode for development builds

## Workflow Integration

### Building JUCE Projects

```bash
# Configure
cmake -B build -DCMAKE_BUILD_TYPE=Release

# Build
cmake --build build --config Release

# Build specific target
cmake --build build --target MyPlugin --config Release
```

2. Build with CMake
3. Test in AudioPluginHost or DAW
4. Debug with IDE (Xcode, Visual Studio, CLion)
5. Iterate
